## Three candidate bar-revision shapes — worked examples

CEO is holding the formal bar-revision call pending these worked
examples. The structural finding (cross-repo CTO-overhead aggregation
multiplier) is real; the question is which mechanism cleanly captures
the anti-absorption intent without opening drift surface.

### Worked example scenarios

To test each candidate, walk it against four cycle shapes:

**Scenario A: this D1 cycle.** 3 repos, 1 engineer + 1 `chore(infra)`
per repo. Aggregate N=6. Per-repo N=2. Engineer share aggregate 3/6 =
50%.

**Scenario B: hypothetical cross-repo cycle with bounded per-repo work
but more shared scaffold.** 3 repos × (1 engineer + 2 scaffold) per repo
= aggregate N=9. Per-repo N=3. Engineer share aggregate 3/9 = 33%.

**Scenario C: hypothetical cross-repo cycle with substantive per-repo
work + one shared scaffold pattern.** 3 repos × (4 engineer + 1
scaffold) per repo = aggregate N=15. Per-repo N=5. Engineer share
aggregate 12/15 = 80%.

**Scenario D: hypothetical cross-repo cycle with large-N per-repo
work.** 3 repos × (10 engineer + 3 scaffold) per repo = aggregate N=39.
Per-repo N=13. Engineer share aggregate 30/39 = 77%.

### Option (a) — enumerate `chore(infra)` (with bounded LOC) in trivial-exclusion list

Effect: removes specific small `chore(*)` patterns from N counting.

| Scenario | N after trivial-exclude | Gate | Verdict |
|---|---|---|---|
| A | aggregate N=3 (only engineer commits remain) | substance gate (N≤5) | PASS |
| B | aggregate N=3 if 2 scaffold commits are all `chore(infra)` ≤LOC; otherwise unchanged | substance OR ratio depending on classification | depends |
| C | aggregate N=12 (1 scaffold per repo excluded) | ratio gate, 12/12 = 100% engineer | PASS |
| D | aggregate N=30 (3 scaffold per repo if all `chore(infra)` ≤LOC; only some likely) | ratio gate, 30/30 = 100% if all excluded; mixed otherwise | likely PASS but depends on `chore(infra)` classification reliability |

**Strengths:** surgical, addresses the exact triggering shape.

**Weaknesses:**
- Fuzzy classification: "infra" could be argued for any scaffold commit. LOC bound helps but is itself a debate surface ("is 5 lines bounded? 10? 20?").
- Doesn't address root cause: cross-repo aggregation. A `feat(scaffold)` cross-repo pattern (legitimately non-trivial) would still flip the boundary at aggregate N>5.
- Drift risk: each new scaffold pattern (chore(deploy)? chore(security)?) becomes a new trivial-exclusion debate.

### Option (b) — substance gate cross-repo when per-repo N ≤ 5

Effect: per-repo N is the scaling parameter for cross-repo cycles.

| Scenario | per-repo N | Gate | Verdict |
|---|---|---|---|
| A | 2 | substance (per-repo ≤5) | PASS via substance |
| B | 3 | substance | PASS via substance |
| C | 5 | substance (boundary) | PASS via substance |
| D | 13 | ratio (per-repo >5) | ratio gate, 30/39 = 77% PASS |

**Strengths:** preserves substance-vs-ratio distinction; addresses
cross-repo aggregation cleanly.

**Weaknesses:**
- Edge case: scenario C (5 per repo × 3 = 15 aggregate) is clearly
  large-N territory but gets substance gate. The per-repo small-N
  threshold doesn't catch this. CEO acknowledged this in their reconsider.
- Anti-creep risk: large multi-repo cycles with 5-or-less per repo could
  use substance gate even when they SHOULD use ratio.

### Option (c) — cross-repo N = max(per-repo N)

Effect: scaling parameter is largest single-repo N.

| Scenario | max per-repo N | Gate | Verdict |
|---|---|---|---|
| A | 2 | substance (max ≤5) | PASS via substance |
| B | 3 | substance | PASS via substance |
| C | 5 | substance (boundary) | PASS via substance |
| D | 13 | ratio (max >5) | per-repo ratio gates per repo, all PASS at 10/13 = 77% per-repo |

**Strengths:** clean mechanical rule, single concept. Cross-repo
cycles where any single repo crosses small-N get the ratio gate
correctly.

**Weaknesses:**
- Same scenario-C edge as option (b): 5 per-repo × 3 repos = max 5 gets
  substance gate but aggregate-15 is clearly large-N territory.
- (c) and (b) behave identically until per-repo crosses 5; the only
  difference is mechanical framing (sum vs max).

### Comparing the options

The scenario-C edge case (5 per-repo × 3 = 15 aggregate getting
substance gate) is the test discriminator:

- **(a)** doesn't hit this edge because `chore(infra)` exclusion is
  pattern-specific. Scenario C with all-engineer commits has aggregate
  N=15 = ratio gate → 12/15 = 80% PASS. Clean.
- **(b)** AND **(c)** would let scenario-C aggregate-15 use substance
  gate, which is the wrong gate for that scale.

But scenario C is hypothetical. **Has a 5-per-repo-each-of-3-repos
cycle ever happened in practice?** Looking at the cycle history:
cycle-1 was single-repo (5 commits engineer + 1 scaffold), cycle-2
single-repo (8 engineer + 3 scaffold), cycle-3 single-repo (1 engineer
+ 4 scaffold after recovery), regex-patch single-repo (1 engineer),
D1 cross-repo (1 engineer + 1 scaffold per repo × 3). No prior cycle
hit the 5-per-repo cross-repo shape.

**The drift surface for option (b)/(c)'s scenario-C edge is hypothetical
in current cycle history.** The drift surface for option (a)'s "infra"
classification debate is also hypothetical until next infra-cross-repo
cycle happens.

### My re-assessed lean: option (c) over my earlier (a)

After working the examples I've shifted. (c)'s mechanical clarity beats
(a)'s pattern-specific surgery:

- **(c) addresses cross-repo aggregation root cause** with one rule
  rather than patch-per-pattern.
- **The scenario-C edge (5-per-repo × multi-repo) is hypothetical;** if
  it surfaces, CEO can amend (c) with anti-creep at THAT cycle's verdict
  rather than pre-emptively complicating the rule now.
- **(c) preserves substance-vs-ratio distinction**; (a) doesn't address
  it (just narrows what counts).
- **(c) preserves the anti-absorption intent** at the per-repo level
  (which is where absorption actually happens) without conflating size
  with shape.

If CEO wants to hedge against the scenario-C drift surface, an explicit
anti-creep clause on (c):

> Cross-repo N = max(per-repo N) for scaling-gate determination.
> Anti-creep: if cross-repo aggregate exceeds N × 2 where N is the
> N-scaling boundary (i.e., aggregate > 10), the ratio gate also
> applies regardless of max per-repo N.

That's belt-and-suspenders: (c) handles the natural case + the
aggregate-explosion case has its own gate.

**My re-assessed lean: option (c) with the anti-creep clause above.**
(Original lean was (a); the scenario-walk shifted me.)

### CEO's earlier lean was (b); my pushback caught the scenario-C edge

CEO acknowledged the scenario-C edge ("5 per-repo × 3 repos = 15
aggregate would still get substance gate, which IS clearly large-N
territory and shouldn't") and reconsidered. (c) with anti-creep
preserves (b)'s structural insight (per-repo is the right granularity)
while closing the scenario-C edge.

## Pragmatic alternative — pattern-revision rather than bar-revision

CEO flagged this in their reconsider message: "revert 3 `chore(infra)`
commits + redo as single consolidated commit in `dexli-family`."
Pattern: family-level conventions live in `dexli-family`, individual
sibling repos inherit.

For the specific `.worktrees/` gitignore case, that would mean:
- One `chore(infra)` commit in dexli-family adding a "common .gitignore
  entries" doc / template
- Each sibling repo's `.gitignore` references the family pattern via
  human discipline + code review
- OR: dexli-family ships a `.gitignore.template` that siblings symlink
  or generate from

This shifts the work from per-repo commits to family-level doc/tooling.
Doesn't help D1 retroactively but is a pattern-shape for future
cross-repo CTO-overhead scaffold work.

**This isn't mutually exclusive with (a)/(b)/(c).** Pattern-revision
addresses the WORK shape (where the CTO commits land); bar-revision
addresses the ATTRIBUTION mechanism (how the counts are evaluated). Both
can land independently.
